## Issues Fixed

### 1. Race condition in createGroup / joinGroup group-count check

**Category:** Race condition
**Files:** `app/(main)/actions.ts`

**Problem:** `MAX_GROUPS_PER_USER` was checked with a `COUNT(*)` query _outside_
the transaction, then the actual insert happened inside it. Two concurrent
`createGroup` calls could both pass the check and both succeed, letting a user
exceed the 10-group limit. `joinGroup` had the same issue â€” the user-level group
count check was outside the transaction (the group-level member count check was
already inside and properly locked).

**Fix:** Moved the user group-count query inside the transaction for both
functions. `createGroup`'s transaction now returns `{ limitReached: true }` when
the limit is hit. `joinGroup`'s transaction checks user group count after the
existing `FOR UPDATE` lock on the group row.

### 2. Dead `selected` field in window ranking insertion

**Category:** Dead code
**File:** `app/(main)/groups/[groupId]/actions.ts`

**Problem:** `generateSchedules()` wrote `selected: i === 0` when inserting
window rankings, but the `selected` column was removed from the `windowRanking`
schema. Additionally, `updateDateConfig()` never included `selected` when
re-inserting rankings, creating an inconsistency.

**Fix:** Removed the `selected: i === 0` property from the `generateSchedules`
insertion.

### 3. Missing price validation across purchase server actions

**Category:** Missing validation
**File:** `app/(main)/groups/[groupId]/schedule/purchase-actions.ts`

**Problem:** `markAsPurchased` accepted `pricePerTicket` and per-assignee
`pricePaid` with zero validation â€” negative numbers or `NaN` could be written to
the database. Other price-accepting functions (`savePurchasePlanEntry`,
`batchSavePurchasePlan`, `updatePurchaseAssigneePrice`, `reportSessionPrice`)
checked for negative values but not `NaN`.

**Fix:** Added `isNaN()` checks alongside existing negative checks in all five
functions. Added both negative and NaN validation to `markAsPurchased` for
`pricePerTicket` and each assignee's `pricePaid`.

### 4. No server-side duplicate purchase prevention

**Category:** Missing server-side guard
**File:** `app/(main)/groups/[groupId]/schedule/purchase-actions.ts`

**Problem:** The UI correctly filters out members who already have a purchase for
a session (via `attendingMemberIds` set). However, `markAsPurchased` had no
server-side check â€” a direct API call could create duplicate purchase records for
the same member + session combination.

**Fix:** Inside the `markAsPurchased` transaction, query existing
`ticketPurchaseAssignee` rows for the session and filter out any assignees who
are already assigned. If all requested assignees are duplicates, return an error.

### 5. No server-side sold-out check in markAsPurchased

**Category:** Missing server-side guard
**File:** `app/(main)/groups/[groupId]/schedule/purchase-actions.ts`

**Problem:** The UI disables the purchase button for sold-out sessions, but
`markAsPurchased` had no server-side check. A direct API call could record a
purchase for a sold-out session.

**Fix:** Inside the `markAsPurchased` transaction, query the `soldOutSession`
table before inserting. If the session is sold out, return an error.

### 6. Preference validation queries outside transactions

**Category:** Race condition (minor)
**File:** `app/(main)/groups/[groupId]/preferences/actions.ts`

**Problem:** `saveBuddies` fetched valid group members and `saveSessionPreferences`
fetched sport rankings and valid sessions _outside_ their transactions. A
concurrent member departure between the query and the transaction could cause
validation against stale data. FK constraints would catch the insert failure, but
the error message would be cryptic.

**Fix:** Moved the validation queries inside their respective transactions. Both
functions now return validation errors via the transaction return value, and the
caller checks for them before returning success.

### 7. ackScheduleWarning fires on save failure in preference wizard

**Category:** UI state bug
**File:** `app/(main)/groups/[groupId]/preferences/_components/preference-wizard.tsx`

**Problem:** In `handlePhaseWarningProceed`, `ackScheduleWarning(group.id)` was
called unconditionally after `await saveAndAdvance()`, even if the save failed.
This cleared the schedule warning on the server despite the user's preferences
not actually being persisted. The user stayed on the current page (correct), but
the warning would not reappear on the next visit.

**Fix:** `saveAndAdvance` now returns a boolean indicating success.
`handlePhaseWarningProceed` only calls `ackScheduleWarning` when the return value
is `true`.

### 8. Sessions loading state uses reference equality

**Category:** Potential stuck UI state
**File:** `app/(main)/groups/[groupId]/preferences/_components/preference-wizard.tsx`

**Problem:** The loading spinner for the sessions step used reference equality
(`sessionsWhenLoadingStarted === sessions`) to detect when the server returned
new data. If the server returned an identical sessions array with a new reference
(common), this works. But if the same reference was returned (e.g., from cache),
the loading spinner could stay forever.

**Fix:** Replaced reference equality with a `{ length, counter }` snapshot. A
monotonic counter is bumped on each load-start, and loading clears when either
`sessions.length` differs from the snapshot or the counter advances.

## Issues Investigated and Determined Not to Fix

### A. getOwnerMembership doesn't filter by active status

**Category:** Defense-in-depth gap
**File:** `lib/auth.ts`

**Finding:** `getOwnerMembership()` queries member by `(groupId, userId)` without
filtering out `pending_approval` or `denied` statuses. In theory, a member with
`role: "owner"` and non-active status could pass the check.

**Why not fixed:** This state cannot occur in practice. Owners are created with
`status: "joined"` and must transfer ownership before leaving (enforced by
`leaveGroup`). No code path sets an owner's status to `denied` or
`pending_approval`. Adding the filter would be harmless but addresses no real
scenario.

### B. removePurchaseAssignee deletes entire purchase when last assignee removed

**Category:** Intentional behavior

**Finding:** When the last assignee is removed from a purchase, the entire
`ticketPurchase` row is deleted.

**Why not fixed:** This is the intended design. Removing all assignees from a
purchase means nobody is attending â€” the purchase record has no purpose. The UI
confirms this action and the user is removing members one by one deliberately.

### C. removePurchaseAssignee returns success for non-existent purchase

**Category:** Low impact

**Finding:** If the purchase was already deleted by another user, the function
returns `{ success: true }` instead of an error.

**Why not fixed:** No data corruption results. The operation is idempotent â€” the
assignee is gone regardless. Adding a "not found" error would create confusing
UX in race scenarios where the page hasn't refreshed yet.

### D. Locked sessions exceeding maxPerDay (3)

**Category:** Intentional design
**File:** `lib/algorithm/combos.ts`

**Finding:** If a member has 4+ purchased (locked) sessions on the same day,
`remainingSlots` goes negative and all locked sessions are placed in a single
combo, exceeding the 3-session-per-day soft limit.

**Why not fixed:** This is the intended design. Purchased tickets must appear in
the schedule regardless of the session count. The 3-session limit is a quality
heuristic for unlocked sessions, not a hard constraint. Users who purchased 4+
sessions for one day made that choice deliberately.

### E. All sessions travel-infeasible on a day = silent schedule loss

**Category:** Intentional design
**File:** `lib/algorithm/combos.ts`

**Finding:** If no subsets for a day are travel-feasible and there are no locked
sessions, the member gets zero combos for that day with no specific warning.

**Why not fixed:** The existing UX flow handles this. If a member is unsatisfied
with their schedule, they know it's because their constraints could not be met
and can update their preferences. Adding per-day granularity to the warning
system would add complexity without changing the user's action (update
preferences).

### F. Locked sessions bypass travel feasibility with no fallback

**Category:** Intentional design
**File:** `lib/algorithm/combos.ts`

**Finding:** If locked sessions alone are travel-infeasible (e.g., two purchased
sessions 30 minutes apart in different zones), the combo is still generated.

**Why not fixed:** Locked sessions represent purchased tickets. The schedule must
include them even if the user will have a tight transfer. The purchase tracker
shows venue/zone information so users can judge feasibility before buying.

### G. Empty travel matrix defaults to 210-minute gaps

**Category:** Non-issue (verified)
**File:** `lib/algorithm/travel.ts`

**Finding:** Missing entries in the travel matrix default to 210 minutes.

**Why not fixed:** Verified that the travel matrix (`scripts/output/driving_times.csv`)
is complete â€” all 17 zones Ă— 17 zones = 289 entries present, no missing values.
The 210-minute default is a safe fallback that would only apply if the data were
corrupted.

### H. Convergence loop can leave violations in final output

**Category:** Known limitation, acceptable

**Finding:** If the convergence loop hits 5 iterations without converging, final
combos may have buddy constraint violations.

**Why not fixed:** Already handled. `nonConvergenceMembers` is tracked and
surfaced as an amber warning in the UI. Adding per-session violation detail would
add significant complexity for marginal benefit.

### I. Empty combos inserted for days with no sessions

**Category:** No downstream impact

**Finding:** If a member has 0 candidate sessions on a day, an empty combo
(`sessionCodes: []`) could theoretically be generated and inserted.

**Why not fixed:** Verified that the insertion code guards against this
(`if (c.sessionCodes.length > 0)`) and the UI filters empty combos
(`.filter(g => g.sessions.length > 0)`). No user-visible impact.

### J. Session filter selections when navigating away from sessions step

**Category:** Already handled

**Finding:** Concern that filter state (sport, type, zone filters) persists when
navigating away from the sessions step.

**Why not fixed:** Verified that `SessionsStep` is conditionally rendered only
when `currentStep === 2`. It unmounts on navigation, destroying all local state
including filter selections. Filters are always reset to empty sets on remount.

### K. Affected buddy notifications and schedule generation blocking

**Category:** Already handled

**Finding:** Concern about whether affected members are properly notified and
schedule generation is blocked after a buddy departs.

**Why not fixed:** Verified the full flow works correctly:

1. `removeMemberTransaction` tracks affected members in `group.affectedBuddyMembers`
2. Error notifications are shown to affected members
3. `generateSchedules` blocks with explicit error if `affectedBuddyMembers` is non-empty
4. Members clear the flag by updating preferences or explicitly confirming review

### L. joinedAt can be null for active members

**Category:** Not possible in practice

**Finding:** `joinedAt` is nullable in the schema.

**Why not fixed:** Verified that every code path to active status sets `joinedAt`:

- `createGroup` sets `joinedAt = now()` for the owner
- `approveMember` sets `joinedAt = now()` when approving
- No path creates a member with `status: "joined"` without setting `joinedAt`

### M. Multi-row edit race in purchase tracker

**Category:** Already handled

**Finding:** Concern about concurrent edits across multiple session rows.

**Why not fixed:** Verified that the `globalBusy` state disables all other rows
when any row is being edited. A `NavigationGuard` also warns when navigating
away during edits. The `beforeunload` handler prevents browser close during edits.

### N. Reported prices orphaned after purchase deletion

**Category:** By design

**Finding:** Reported prices are not cleaned up when a purchase is deleted.

**Why not fixed:** Reported prices and purchases are independent concepts.
A price report is informational ("this session costs $X") and remains useful
regardless of whether a specific purchase record exists.

### O. Empty group after last member leaves

**Category:** Not possible

**Finding:** Concern about zombie groups with no active members.

**Why not fixed:** The owner must transfer ownership before leaving. If there's
only one member, they must be the owner and cannot leave â€” the only option is
deleting the group. The code enforces this at `leaveGroup` (line 394-398).

### 9. Missing database indexes on high-frequency queries

**Category:** Performance
**File:** `lib/db/schema.ts`

**Problem:** `windowRanking.groupId` and `purchaseTimeslot.groupId` had no indexes.
Both columns are filtered on every group detail page load (`getGroupDetail` queries
`windowRanking WHERE groupId` and `purchaseTimeslot WHERE groupId`). Without indexes,
these become full table scans as groups accumulate data.

**Fix:** Added `window_ranking_group_idx` on `windowRanking.groupId` and
`purchase_timeslot_group_idx` on `purchaseTimeslot.groupId`. Migration `0025`.

### 10. No error handling on confirmAffectedBuddyReview

**Category:** Missing error handling
**Files:** `app/(main)/groups/[groupId]/preferences/_components/preference-wizard.tsx`,
`app/(main)/groups/[groupId]/preferences/_components/review-step.tsx`

**Problem:** `handleConfirmReview` in the preference wizard called
`confirmAffectedBuddyReview()` without checking the return value or catching errors.
If the server action failed (network error, auth error, transaction error), the user
saw no error message â€” the button spinner cleared and nothing happened. The
affected-buddy flag remained set server-side, but the user believed they had confirmed.

**Fix:** `handleConfirmReview` now checks `result.error` and sets the error state
displayed in the wizard. Added try/catch for network errors. `ReviewStep`'s button
handler uses try/finally to ensure `setConfirming(false)` always runs.

### 11. Group schedule loading state clears before error is set

**Category:** UI state bug
**File:** `app/(main)/groups/[groupId]/group-schedule/group-schedule-content.tsx`

**Problem:** In `fetchSchedule`, `setLoading(false)` ran immediately inside `.then()`

rendered â€” the user saw a blank state with no loading indicator. On retry, there
was no visual feedback that loading had started because loading was already false.

**Fix:** Moved `setLoading(false)` into `.finally()` so it runs after success or
error handling. Added `.catch()` for unhandled network errors.

### 12. Buddies number input accepts negative characters momentarily

**Category:** UX polish
**File:** `app/(main)/groups/[groupId]/preferences/_components/buddies-step.tsx`

**Problem:** The `minBuddies` number input allowed typing `-`, `e`, `E`, and `+`
characters. The onChange handler clamped values correctly, but the invalid characters
appeared briefly in the input before being replaced on the next event cycle.

**Fix:** Added `onKeyDown` handler that calls `e.preventDefault()` for `-`, `e`,
`E`, and `+` keys, preventing the characters from appearing at all.

### 13. Nullable pricePerTicket undocumented

**Category:** Documentation
**File:** `lib/db/schema.ts`

**Problem:** Migration `0024` changed `ticketPurchase.pricePerTicket` from required
to nullable, but the schema had no comment explaining why the field is nullable.

**Fix:** Added inline comment: "Nullable: price may be unknown at time of purchase
recording."

## Second Audit â€” Issues Investigated and Determined Not to Fix

### P. Optimistic UI updates in purchase tracker don't roll back on failure

**Category:** Already handled

**Finding:** Concern that `onSessionOverride` updates local state optimistically
and server rejection leaves the UI showing false state.

**Why not fixed:** Verified that all purchase tracker server actions (markAsSoldOut,
markAsPurchased, reportSessionPrice, etc.) properly check `result.error` and display
errors to the user. The optimistic state is overwritten on the next `router.refresh()`
or page navigation. The `globalBusy` state prevents concurrent edits, so partial
state corruption doesn't occur.

### Q. Cascading interest count drops cause unexpected session elimination

**Category:** Intentional design
**File:** `lib/algorithm/runner.ts`

**Finding:** When sessions are pruned in iteration N, iteration N+1 rebuilds
interest counts. A session with 3 interested members can drop to 2, triggering
minBuddies violations and further cascading pruning.

**Why not fixed:** This is the intended convergence mechanism. The algorithm
deliberately re-evaluates constraints after each pruning pass to find a stable
solution. If cascading eliminates too many sessions, the member lands in
`membersWithNoCombos` or `nonConvergenceMembers`, both of which are surfaced in the
UI with clear guidance to adjust preferences.

### R. Fairness penalty in window ranking scales quadratically with group size

**Category:** Intentional design
**File:** `lib/algorithm/window-ranking.ts`

**Finding:** The fairness penalty formula `stdev Ă— memberCount Ă— 0.5` scales with
member count, making the penalty 3x higher for 12 members than for 4 members with
the same stdev. Can produce negative window scores.

**Why not fixed:** This is the intended design. Larger groups should be penalized
more heavily for unfair score distributions because unfairness affects more people.
The penalty ensures the top-ranked windows are fair across all members, not just
high-scoring for a subset.

### S. Session dirty detection in SessionsStep only checks map size

**Category:** Not a bug (verified)
**File:** `app/(main)/groups/[groupId]/preferences/_components/sessions-step.tsx`

**Finding:** The mount sync effect uses `preferences.size !== initialPreferences.size`
to detect if filtering changed the preferences.

**Why not fixed:** Verified that the filtering logic (lines 43â€“50) can only _remove_
entries from `initialPreferences` â€” it never adds or modifies entries. If no entries
are removed, size matches AND content is identical. If entries are removed, size
decreases and the sync fires. The size check is sufficient.

### T. Navigation guard blocks silently with no user feedback

**Category:** Already handled
**File:** `app/(main)/groups/[groupId]/_components/navigation-guard-context.tsx`

**Finding:** Concern that `GuardedLink` prevents navigation with no visual feedback.

**Why not fixed:** Verified that a full "Unsaved Changes" modal is displayed via
`NavigationGuardContext`. The modal names the dirty steps (e.g., "Buddies",
"Sport Rankings"), and offers "Stay" and "Discard & Leave" buttons. Browser
back/forward is also intercepted with the same modal. Browser refresh/close uses
the native `beforeunload` prompt.

### U. Hard buddy filter silently ignores departed member IDs

**Category:** Already handled
**File:** `lib/algorithm/filter.ts`

**Finding:** If a hard buddy ID references a member no longer in the group, the
filter silently skips that buddy's constraint.

**Why not fixed:** Verified the full departure flow handles this correctly:

1. `removeMemberTransaction` adds affected members to `group.affectedBuddyMembers`
2. Notifications display departed buddy names with guidance to update preferences
3. `saveBuddies` validates buddy IDs inside its transaction against current active

**Category:** Intentional design
**File:** `app/(main)/groups/[groupId]/purchase-tracker/purchase-tracker-content.tsx`

**Finding:** When editing any session row, all other session rows are disabled.

**Finding:** Concern that purchases and reported prices are deleted in separate

### X. Resilience comparison in window ranking lacks floating-point tolerance

## Third Audit â€” Additional Fixes

### 14. Status promotion bug in removeMemberTransaction

**Category:** Logic bug
**File:** `app/(main)/groups/[groupId]/actions.ts`

**Problem:** When a departing member was part of the last schedule, the batch
status reset targeted all active members via
`notInArray(member.status, ["pending_approval", "denied"])`. This incorrectly
promoted members with status `"joined"` (who had not yet completed the
preference wizard) to `"preferences_set"`. These members then appeared as
"ready" in the generate-schedule section despite having no sport rankings, no
session preferences, and no buddy constraints. If the owner generated schedules,
the algorithm produced 0 combos for the promoted member, leaving the group stuck.

**Fix:** Changed the WHERE clause to `eq(member.status, "preferences_set")` so
only members who genuinely completed preferences get the `statusChangedAt`
timestamp bump. Members still at `"joined"` are unaffected by schedule deletion
(they had no combos anyway) and remain correctly shown as "not ready."

### 15. Race condition in denyMember

**Category:** Race condition
**File:** `app/(main)/groups/[groupId]/actions.ts`

**Problem:** `denyMember` read the member's status outside any transaction,
checked it was `"pending_approval"`, then ran an UPDATE with a WHERE clause
matching only `member.id`. If a concurrent `approveMember` changed the status to
`"joined"` between the check and the update, the UPDATE would overwrite the
approval â€” silently denying an already-approved member.

**Fix:** Added `eq(member.status, "pending_approval")` to the UPDATE WHERE
clause. If a concurrent operation already changed the status, the UPDATE is a
no-op instead of overwriting the new status.

### 16. Overview tab badge gap for post-generation joiners

**Category:** UI logic gap
**File:** `app/(main)/groups/[groupId]/_components/group-shell.tsx`

**Problem:** `schedulesNeedAttention` checked for newly joined members via
`status === "joined"` only. A member who joined after generation AND completed
their preferences (becoming `"preferences_set"`) fell through both the
`hasNewlyJoinedMembers` check (wrong status) and `hasUpdatedPrefs` check
(explicitly excluded post-gen joiners). The Overview tab badge disappeared even
though regeneration was needed, while the notifications section and generate
button correctly identified the need.

**Fix:** Added a `hasNewMembersWithPrefs` check that catches members with
`status === "preferences_set"` and `joinedAt > scheduleGeneratedAt`, matching
the logic already used in `generate-schedule-section.tsx`.

### 17. Missing fetch race guard in group-schedule-content

**Category:** Defensive improvement
**File:** `app/(main)/groups/[groupId]/group-schedule/group-schedule-content.tsx`

**Problem:** `fetchSchedule()` had no request-ordering guard. If the effect
fired twice in succession (e.g., due to rapid dependency changes), responses
could arrive out of order, causing stale data to overwrite fresh data.
`schedule-content.tsx` already used a `fetchCounterRef` pattern for this.

**Fix:** Added the same `fetchCounterRef` pattern: each fetch increments a
counter, and callbacks ignore responses from superseded requests.

### 18. API route returns denied members

**Category:** Inconsistency
**File:** `app/api/groups/[groupId]/route.ts`

**Problem:** The member query did not filter out denied members, unlike
`getGroupDetail.ts` which uses `notInArray(member.status, ["denied"])`.
Consumers of the API route saw a different member list than server-rendered
pages.

**Fix:** Added `notInArray(member.status, ["denied"])` to the member query for
consistency with `getGroupDetail`.

## Fourth Audit â€” Additional Fixes

### 19. Race condition in approveMember (symmetric to denyMember fix #15)

**Category:** Race condition
**File:** `app/(main)/groups/[groupId]/actions.ts`

**Problem:** `approveMember` read the member's status outside the transaction,
checked it was `"pending_approval"`, then ran an UPDATE with a WHERE clause
matching only `member.id`. If a concurrent `denyMember` changed the status to
`"denied"` between the check and the update, the UPDATE would overwrite the
denial â€” silently approving an already-denied member. Additionally, the
`approvedUser` SELECT after the UPDATE had no status filter, so the
`departedMembers.rejoinedAt` logic would fire even when the UPDATE was a no-op,
creating inconsistent data.

**Fix:** Added `eq(member.status, "pending_approval")` to the UPDATE WHERE
clause (matching the pattern already applied to `denyMember` in fix #15). Also
added `eq(member.status, "joined")` to the subsequent `approvedUser` SELECT so
the departed-member rejoin logic only runs when the approval actually took
effect. On concurrent status change, the UPDATE is a 0-row no-op, the
`approvedUser` SELECT returns empty, rejoin logic is skipped, and the function
returns success (idempotent â€” the UI refreshes to show the actual state).

### 20. Race condition in transferOwnership (group left ownerless)

**Category:** Race condition
**File:** `app/(main)/groups/[groupId]/actions.ts`

**Problem:** `transferOwnership` verified the target member's existence and
status outside the transaction, then performed two role UPDATEs inside the
transaction with no `FOR UPDATE` lock. If the target member departed via
`leaveGroup` or was removed via `removeMember` between the pre-check and the
transaction, the first UPDATE (owner â†’ member) would succeed but the second
(target â†’ owner) would be a no-op on the deleted row. The group would be left
with no owner â€” a permanently broken state.

**Fix:** Added a `FOR UPDATE` lock on the group row inside the transaction
(serializing with `leaveGroup`/`removeMember` which also lock this row). After
acquiring the lock, the target member is re-verified: if the member no longer
exists or has an inactive status (`pending_approval`/`denied`), the transaction
returns early without modifying any roles. The caller returns `MSG_MEMBER_NOT_FOUND`.

### 21. Race condition in joinGroup denied re-request

**Category:** Race condition (defensive)
**File:** `app/(main)/actions.ts`

**Problem:** When a denied member re-requested to join via the same invite code,
the UPDATE resetting status from `"denied"` to `"pending_approval"` used a WHERE
clause matching only `member.id`. If a concurrent operation changed the member's
status between the pre-check and the UPDATE, the new status would be silently
overwritten. While the group row `FOR UPDATE` lock inside the transaction
makes this race extremely unlikely, the pattern was inconsistent with the
defensive guard already applied to `denyMember` (fix #15).

**Fix:** Added `eq(member.status, "denied")` to the UPDATE WHERE clause.
If the status changed concurrently, the UPDATE is a 0-row no-op. The function
returns success (idempotent â€” the UI refreshes to show the actual state).

### 22. Generate-schedule button tooltip misleading when blocked by buddy reviews

**File:** `app/(main)/groups/[groupId]/_components/generate-schedule-section.tsx`

**Problem:** The generate-schedule button's disabled tooltip always showed
"All members must set their preferences first." regardless of the actual reason
the button was disabled. When the blocker was affected buddy reviews or
no-combo members who hadn't updated preferences, the message was incorrect and
confusing â€” the owner would see all members listed as having set preferences

- No-combo members haven't updated â†’ "Members without sessions need to update

- Members missing preferences â†’ "All members must set their preferences first."
  (unchanged fallback)

### 23. removeMembership DELETE missing status guard

**Category:** Race condition
**File:** `app/(main)/actions.ts`

**Problem:** `removeMembership` (used to cancel a pending join request or
remove a denied membership) checked the member's status outside the DELETE
(`pending_approval` or `denied`), then ran a bare DELETE with a WHERE clause
matching only `member.id`. If a concurrent `approveMember` changed the status to
`"joined"` between the check and the DELETE, the DELETE would remove an active
member â€” silently deleting someone who was just approved.

**Fix:** Added `inArray(member.status, ["pending_approval", "denied"])` to the
DELETE WHERE clause. If the status changed concurrently, the DELETE matches 0
rows (no-op). The function returns success (idempotent â€” the user's page
refreshes to show the group now appears in "Joined Groups" instead of
"Pending Requests").

### 24. leaveGroup / removeMember race with concurrent ownership transfer

**Category:** Race condition
**Files:** `app/(main)/groups/[groupId]/actions.ts`

**Problem:** Both `leaveGroup` and `removeMember` checked the target member's
role outside the transaction (`role === "owner"` guard), then called
`removeMemberTransaction` inside a transaction. If a concurrent
`transferOwnership` completed between the pre-check and the transaction's
`FOR UPDATE` lock â€” making the departing member the new group owner â€” the
transaction would delete the owner, leaving the group permanently ownerless.

Both `transferOwnership` and `removeMemberTransaction` lock the group row with
`FOR UPDATE`, so they serialize at the lock. But the role pre-check used stale
data from before the lock was acquired. Sequence: (1) pre-check sees
`role = "member"` â†’ passes, (2) concurrent transfer makes member the owner,
(3) transaction acquires lock, (4) `removeMemberTransaction` deletes the
now-owner member.

**Fix:** Added `role: member.role` to the existing departing-user SELECT inside
`removeMemberTransaction` (which runs after the `FOR UPDATE` lock is acquired),
then check `departingUser?.role === "owner"`. If true, throw an error that
aborts the entire transaction â€” rolling back any buddy constraint or session
preference deletes that occurred earlier in the transaction. Both callers
(`leaveGroup` and `removeMember`) catch the specific error and return a
descriptive message to the UI.

### AF. reportedPrice allows multiple reports per member per session
