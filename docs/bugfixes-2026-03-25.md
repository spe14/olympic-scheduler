# Bug Fixes — 2026-03-25

## Preference wizard: generic "Loading..." flash during step transitions

**Symptom**: When moving between preference steps with the phase warning dialog (schedule already generated + dirty data), the step-specific loading message (e.g., "Loading sessions...") would sometimes get replaced by the generic "Loading..." from `app/(main)/groups/[groupId]/loading.tsx`.

**Root cause**: In `handlePhaseWarningProceed`, the `ackScheduleWarning` server action was called fire-and-forget (not awaited). When its response arrived asynchronously, the `revalidatePath` it contains was processed outside a React transition, triggering the Suspense boundary fallback.

**History**: Originally (c605aa3), the ack was awaited _before_ the save. The code review fix (14f17cf) reversed the order to save-first-then-ack — preventing the warning from being cleared when the save failed — but made the ack fire-and-forget as a side effect.

**Fix**: Await `ackScheduleWarning` after `saveAndAdvance` completes. The save-first ordering is preserved, and awaiting ensures the revalidation is processed within a React transition (no Suspense fallback flash). The step has already advanced and `setSaving(false)` has already been called, so the user sees no delay.

**File**: `app/(main)/groups/[groupId]/preferences/_components/preference-wizard.tsx` — `handlePhaseWarningProceed`

**Revert**: Change `await ackScheduleWarning(group.id)` back to `ackScheduleWarning(group.id)` in `handlePhaseWarningProceed`.

## Preference wizard: "Loading..." flash on first sport rankings save

**Symptom**: On the first time saving sport rankings and advancing to the session interests step (no prior rankings/session data), the "Loading sessions..." message was briefly replaced by the generic "Loading..." from `app/(main)/groups/[groupId]/loading.tsx`.

**Root cause**: Any server action (`saveSportRankings`) triggers a re-render of the page's server component. The nearest Suspense boundary — `app/(main)/groups/[groupId]/loading.tsx` — catches the suspension and flashes "Loading...". This only happens on the first save because subsequent visits skip the server action entirely (`completedSteps.has(currentStep) && !isCurrentStepDirty()` early return in `saveCurrentStep`).

**Why `revalidatePath` alone didn't fix it**: Adding `revalidatePath` to `saveSportRankings` bundles the new RSC tree with the action response, but the `[groupId]/loading.tsx` Suspense boundary still triggers during the page re-render regardless.

**Fix**: Added `revalidatePath` to `saveSportRankings` so the updated sessions data is included in the server action response (faster transition). This reduces the flash duration but does not fully eliminate it — the `[groupId]/loading.tsx` Suspense boundary still briefly triggers during the page re-render.

- **File**: `app/(main)/groups/[groupId]/preferences/actions.ts` — `saveSportRankings`, before `return { success: true }`
- **Revert**: Remove the `revalidatePath(`/groups/${groupId}`, "layout")` line from `saveSportRankings`.

**Not pursued**: A preferences-specific `loading.tsx` returning `null` was tried but reverted — it replaced the "Loading..." flash with a blank screen flash, which was worse.

## Off-schedule session card: inline edits/deletes didn't disable other cards

**Symptom**: In the Purchase Tracker's off-schedule sessions section, clicking the delete button on a reported price or the edit/delete buttons on a purchase assignee price did not disable buttons on other session cards. This was inconsistent with the on-schedule `TrackerSessionCard`, which disables all other cards' interactive elements during inline operations.

**Root cause**: `OffScheduleSessionCard` did not have access to `onSetGlobalBusy` — it only received a read-only `busy` prop from its parent. Without signaling global busy, its `isPending` state (from `useTransition`) was local to the card and invisible to siblings. The on-schedule `TrackerSessionCard` had this wired correctly via `localBusy` + `useEffect` sync + `onSetGlobalBusy`.

**Fix**: Threaded `onSetGlobalBusy` from `OffSchedulePurchase` through to `OffScheduleSessionCard`. Added:

1. `localBusy` state (`isPending || editingAssignee !== null`) matching `TrackerSessionCard` pattern
2. `useEffect` that calls `onSetGlobalBusy(false)` when `localBusy` goes false (cleanup sync)
3. `onSetGlobalBusy(true)` calls in the reported price delete handler, purchase price edit handler, and purchase assignee delete handler

**File**: `app/(main)/groups/[groupId]/purchase-tracker/purchase-tracker-content.tsx` — `OffScheduleSessionCard`

**Revert**: Remove `onSetGlobalBusy` prop from `OffScheduleSessionCard`, remove the `localBusy` / `useEffect` block, remove `onSetGlobalBusy(true)` calls from the three handlers, and revert `busy` computation to `externalBusy || isPending || editingAssignee !== null`.

## Purchase tracker: stale OOB/sold-out badge after regeneration

**Symptom**: Mark a session as out of budget on the purchase tracker, regenerate (session moves to excluded), undo the OOB mark from excluded sessions, regenerate again (session returns to scheduled sessions). The session correctly appears on the schedule but still shows the "Out of Budget" badge.

**Root cause**: The purchase tracker uses `sessionOverrides` (client-side optimistic state) keyed by session code. When the user marked OOB, the handler called `onOverride({ outOfBudget: true })`, writing into `sessionOverrides`. After regeneration, `fetchData` fetched fresh server data (with `isOutOfBudget: false`), but `sessionOverrides` was never cleared. The override took precedence: `overrides?.outOfBudget ?? s.isOutOfBudget` resolved to `true` from the stale override.

**Why only OOB and not sold-out**: Sold-out marks from the excluded sessions section use a separate `locallyUnmarked` state in the `ExcludedSessions` component, not `sessionOverrides`. The sold-out toggle on on-schedule cards does use `sessionOverrides`, but that scenario wasn't tested (mark sold out on-schedule → regenerate → undo from excluded → regenerate).

**Fix**: Clear `sessionOverrides` in `fetchData` when fresh server data arrives: `setSessionOverrides(new Map())`. Fresh server data is authoritative — any override that was persisted is reflected in the server response; any that wasn't should not persist.

**File**: `app/(main)/groups/[groupId]/purchase-tracker/purchase-tracker-content.tsx` — `fetchData`

**Revert**: Remove the `setSessionOverrides(new Map())` line from `fetchData`.
