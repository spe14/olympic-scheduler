# Bug Fixes — 2026-03-22

Code review identified several functional issues across the codebase. All fixes have been applied and verified against the full test suite (1125 tests passing).

## Fix 1: Timezone-Safe Olympic Day Generation

**Problem:** `new Date("2028-07-12")` in `generateSchedules` parses as UTC midnight. When `setDate()`/`getDate()` operate in the server's local timezone and `toISOString()` converts back to UTC, dates can drift by one day on servers west of UTC (e.g., US timezones). The same issue existed in `getDateDisplay` with `T00:00:00`.

**Fix:** Changed to `new Date("2028-07-12T12:00:00")` — noon prevents day drift regardless of timezone. Matches the pattern already used in `window-ranking.ts` and schedule content files.

**Files changed:**

- `app/(main)/groups/[groupId]/actions.ts` — Olympic day generation loop
- `lib/utils.ts` — `getDateDisplay` date parsing

## Fix 2: Centralized PT Time Formatters

**Problem:** Duplicated `formatTime`, `formatDate`, `formatTimestamp` functions across 6+ component files with no timezone awareness. Session times should display in Pacific Time (LA 2028 Olympics); only user action timestamps should use browser local time.

**Fix:** Added 5 shared formatters to `lib/utils.ts`:

- `formatSessionTime(timeStr)` — "HH:MM" to "H:MM AM/PM" (pure string parsing)
- `formatSessionDate(dateStr)` — "YYYY-MM-DD" to "Wed, Jul 12" in PT
- `formatSessionDateHeader(dateStr)` — `{ weekday, monthDay }` in PT for calendar headers
- `formatTimeslotDateTime(date)` — UTC timestamp formatted in PT with "PT" suffix
- `formatActionTimestamp(date)` — UTC timestamp in browser local time

Replaced local formatters in 5 component files. Updated timeslot form to interpret all date/time input as Pacific Time using `Intl.DateTimeFormat` with `timeZone: "America/Los_Angeles"`.

**Files changed:**

- `lib/utils.ts` — shared formatters
- `schedule/_components/session-detail-modal.tsx` — removed 3 local formatters
- `schedule/schedule-content.tsx` — removed 2 local formatters
- `group-schedule/group-schedule-content.tsx` — removed 3 local formatters
- `purchase-tracker/purchase-tracker-content.tsx` — removed 3 local formatters
- `_components/notifications-section.tsx` — removed 1 local formatter
- `schedule/_components/timeslot-form.tsx` — PT-aware date/time input handling

## Fix 3: Locked (Purchased) Sessions Always Appear on Schedule

**Problem:** Purchased sessions could be silently dropped from a member's schedule during regeneration at three points in the algorithm pipeline:

1. **Not in preferences:** `candidateSessions` was built exclusively from `sessionPreference` rows. Off-schedule purchases (tickets bought for the member by someone else) with no preference row were silently excluded.
2. **Filtered by constraints:** `filterCandidateSessions` applied hard buddy and minBuddies filters to all sessions including locked ones. A locked session failing these constraints was removed before combo generation.
3. **Pruned by convergence loop:** The convergence loop pruned sessions that generated post-generation violations. Locked sessions failing buddy checks were pruned on the next iteration.

**Fix (4 changes):**

- `actions.ts` (generateSchedules) — Query the `session` table for locked codes missing from preferences, inject them into `candidateSessions` with `interest: "high"`
- `filter.ts` (filterCandidateSessions) — Separate locked sessions before filtering, apply hard buddy and minBuddies filters only to unlocked sessions, return `[...locked, ...filteredUnlocked]`
- `runner.ts` (convergence loop) — Skip locked session codes when pruning violating sessions
- `validation.ts` (validatePostGeneration) — Skip buddy/minBuddies validation on locked sessions to avoid wasting convergence iterations on unresolvable violations

**Guarantee:** With these four changes, locked sessions survive through to `generateDayCombos` in `combos.ts`, which already handles them correctly — they are required in every combo, and if nothing is travel-feasible alongside them, the code falls back to a locked-only combo. Locked sessions appear on the final schedule regardless of travel constraints, >3 sessions per day, or buddy constraint violations.

**Files changed:**

- `app/(main)/groups/[groupId]/actions.ts` — inject missing locked sessions
- `lib/algorithm/filter.ts` — exempt locked from constraint filters
- `lib/algorithm/runner.ts` — never prune locked from candidates
- `lib/algorithm/validation.ts` — skip validation on locked sessions

## Fix 4: Non-Atomic Purchase Mutations

**Problem:** `markAsSoldOut`, `unmarkSoldOut`, `markAsOutOfBudget`, and `unmarkOutOfBudget` each performed two separate DB operations (entity insert/delete + `purchaseDataChangedAt` timestamp update) outside a transaction. If the timestamp update failed, the entity was modified but the regeneration notification would not fire.

**Fix:** Wrapped each function's DB operations in `db.transaction()`.

**File changed:**

- `app/(main)/groups/[groupId]/schedule/purchase-actions.ts`

## Fix 5: JSONB Race Conditions with Row Locking

**Problem:** Several functions performed read-modify-write on JSONB fields (`affectedBuddyMembers`, `departedMembers`) without proper locking. Concurrent updates from different members could cause lost updates (last-write-wins).

**Fix:** Added `.for("update")` (Drizzle ORM's `SELECT ... FOR UPDATE`) to the group row select within each transaction, preventing concurrent modifications:

| Function                     | File                     | Change                                                                     |
| ---------------------------- | ------------------------ | -------------------------------------------------------------------------- |
| `confirmAffectedBuddyReview` | `preferences/actions.ts` | Wrapped in `db.transaction`, added `.for("update")`                        |
| `approveMember`              | `actions.ts`             | Wrapped departedMembers update in `db.transaction`, added `.for("update")` |
| `saveBuddies`                | `preferences/actions.ts` | Added `.for("update")` to existing transaction                             |
| `saveSessionPreferences`     | `preferences/actions.ts` | Added `.for("update")` to existing transaction                             |
| `removeMemberTransaction`    | `actions.ts`             | Added `.for("update")` to existing transaction                             |

**Files changed:**

- `app/(main)/groups/[groupId]/actions.ts`
- `app/(main)/groups/[groupId]/preferences/actions.ts`

## Fix 6: `deletePurchase` Missing groupId Scope

**Problem:** The verification query in `deletePurchase` checked `ticketPurchase.id` but not `ticketPurchase.groupId`. A user in Group A could verify the existence of a purchase ID from Group B (the actual delete was properly scoped, but the auth check leaked information).

**Fix:** Added `eq(ticketPurchase.groupId, groupId)` to the verification query WHERE clause.

**File changed:**

- `app/(main)/groups/[groupId]/schedule/purchase-actions.ts`

---

## Fix 7: Floating-Point Score Comparison

**Problem:** Combo sorting used exact `!==` comparison on floating-point scores (`b.score !== a.score`). Accumulated rounding errors from the scoring formula (sport multiplier x session adjustment x soft buddy bonus) could cause two effectively-equal scores to compare as different, leading to non-deterministic combo ranking.

**Fix:** Changed to epsilon comparison: `Math.abs(b.score - a.score) > 1e-9`.

**File changed:**

- `lib/algorithm/combos.ts`
